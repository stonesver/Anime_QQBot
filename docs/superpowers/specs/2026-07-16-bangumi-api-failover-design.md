# Bangumi API 自动故障转移设计

## 背景与目标

生产服务器无法稳定连接 `https://api.bgm.tv`，但可以访问由运维人员验证过的兼容镜像。系统需要保留官方 API 为首选，并在临时网络或上游故障时自动切换到通过环境变量配置的备用地址。

本变更只处理 Bangumi API 客户端的地址选择，不改变 `bangumi-data` 同步、数据库缓存或通知调度行为。

## 配置

新增两个配置项：

```dotenv
BANGUMI_API_BASE_URL=https://api.bgm.tv
BANGUMI_API_FALLBACK_URLS=https://bgmapi.anibt.net,https://api.bangumi.lol
```

- `BANGUMI_API_BASE_URL` 是主地址，默认值为官方 API。
- `BANGUMI_API_FALLBACK_URLS` 是按优先级排列、以逗号分隔的备用地址，默认值为空。
- 配置加载时去除地址两侧空白和结尾 `/`，忽略空项，并拒绝不使用 `http` 或 `https` 的地址。
- 重复地址只保留第一次出现的位置；与主地址重复的备用项会被忽略。
- 项目不在代码中内置或背书任何第三方镜像；部署者自行选择备用地址。

失败冷却时间固定为 300 秒。本期不增加对应环境变量，避免扩大配置面；以后有实际运维需求时再开放。

## 请求与故障转移

`BangumiClient` 保存有序端点列表和每个端点的进程内冷却截止时间。每个请求按以下规则执行：

1. 按主地址、备用地址的顺序选择当前未冷却的端点。
2. 若所有端点都在冷却中，选择最早结束冷却的端点进行一次探测，避免客户端完全停摆。
3. 请求成功后立即返回，并清除该端点的失败冷却状态。
4. 网络连接异常、超时、HTTP `429`、HTTP `5xx` 或无法解析 JSON 时，将当前端点冷却 300 秒并尝试下一个端点。
5. HTTP `4xx`（`429` 除外）属于请求、权限或资源问题，不切换端点，直接保留现有永久错误语义。
6. 所有候选端点失败后，抛出最后一个提供方错误；上层现有同步状态和数据库快照降级机制继续生效。

冷却状态仅保存在 worker 进程内，不写数据库。当前只有单个目录同步 worker，因此无需引入分布式健康状态。

## 访问令牌隔离

`BANGUMI_ACCESS_TOKEN` 只发送给 `BANGUMI_API_BASE_URL`。请求备用地址时不携带 `Authorization` 头，防止将用户令牌泄露给第三方镜像。

`User-Agent` 和 `Accept: application/json` 会发送给所有端点。搜索、详情、日历和剧集请求均使用同一套故障转移逻辑。

## 组件改动

- `Settings`：解析主地址和备用地址列表，提供经过验证的有序配置。
- `run_worker`：将地址配置传给 `BangumiClient`。
- `BangumiClient`：负责端点选择、失败分类、冷却和令牌隔离。
- `.env.example` 与部署文档：说明默认值、备用地址格式及第三方镜像风险。

不新增数据库表、迁移或后台管理命令。

## 可观测性与错误处理

端点发生切换时记录警告日志，包含失败端点、错误种类和即将尝试的端点，但不记录访问令牌、完整请求体或其他秘密。

提供方同步状态仍使用现有 `catalog_sync_states` 记录最终成功或最终失败。单个端点失败但备用端点成功时，整个 Bangumi 同步应记录为成功。

## 测试与验收

自动化测试至少覆盖：

- 未配置备用地址时仍请求官方默认地址；
- 主地址连接失败、超时、`429`、`5xx` 或无效 JSON 时按顺序切换；
- 永久 `4xx` 不触发切换；
- 失败地址在冷却期内被跳过，冷却结束后恢复优先探测；
- 所有端点都在冷却中时仍选择一个端点探测；
- 主地址收到访问令牌，备用地址不收到访问令牌；
- 备用地址环境变量正确处理逗号、空白和空项，无效 URL 被拒绝；
- 现有 Bangumi 映射与错误分类测试继续通过。

验收命令包括完整测试套件、类型检查、代码格式检查及 `docker compose config --quiet`。

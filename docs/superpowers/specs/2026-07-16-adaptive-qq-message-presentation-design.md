# QQ 命令自适应消息呈现设计

日期：2026-07-16

## 1. 背景

当前查询、详情和帮助命令主要以连续纯文本返回。番剧名称、Bangumi ID、预计放送时间和时区挤在同一行，列表稍长就难以扫读；帮助命令也缺少清晰的命令分组。

项目已经具备 `OutboundMessage.markdown`、消息按钮和番剧封面 URL，但现有命令渲染器尚未充分使用这些能力。QQ 群聊与单聊现已支持自定义 Markdown 和自定义按钮，无需单独申请模板：

- [QQ Markdown 消息](https://bot.q.qq.com/wiki/develop/api-v2/server-inter/message/type/markdown.html)
- [QQ 消息按钮](https://bot.q.qq.com/wiki/develop/api-v2/server-inter/message/trans/msg-btn.html)
- [QQ 消息发送](https://bot.q.qq.com/wiki/develop/api-v2/server-inter/message/send-receive/send.html)

## 2. 目标与非目标

### 2.1 目标

- 根据命令场景和结果数量自动选择图文卡、结构化 Markdown 或紧凑文本。
- 建立统一的信息层级，突出标题和放送时间，弱化内部 ID 与时区等次要信息。
- 为长结果提供可直接点击的分页交互。
- 在封面缺失、图片加载失败或 Markdown 被拒绝时保留完整可读的降级结果。
- 保持领域层与 QQ 展示格式解耦。

### 2.2 非目标

- 本轮不改变番剧查询、订阅、权限或计划任务的业务规则。
- 本轮不美化定时推送与开播提醒；它们可在后续复用本设计的展示策略。
- 不引入海报生成、截图渲染、对象存储或新的外部图片服务。
- 不使用需要额外申请权限的被动 ARK 模板。

## 3. 设计决策

采用统一自适应渲染器，而不是为每个命令复制独立模板，也不把模板规则外置到 YAML 或 JSON。

展示策略同时接收场景和安全过滤后的实际结果数量，返回以下模式之一：

| 实际条数 | 展示模式 | 说明 |
| --- | --- | --- |
| 1–20 | `CARD` | 原生 Markdown 图文卡 |
| 21–50 | `STRUCTURED` | 按日期或星期分组的结构化 Markdown |
| 51 及以上 | `COMPACT` | 紧凑分页文本 |

场景可以覆盖数量规则：

- `番剧详情`、`下次更新` 固定为单番图文卡；
- `帮助` 固定为分组式结构化 Markdown；
- `订阅`、`取消订阅` 固定为轻量状态卡；
- 参数错误、权限错误、无结果固定使用简短文本。

NSFW 条目必须先过滤，再计算用于选择模式的数量。

## 4. 架构与模块边界

命令处理层继续取得结构化的番剧与放送数据。展示层负责模式选择、分页、标题层级、时间格式、Markdown 转义和按钮布局。

```text
命令取得结构化数据
        ↓
PresentationPolicy 选择模式
        ↓
Card / Structured / Compact Renderer
        ↓
OutboundMessage（纯文本兜底 + Markdown + 按钮）
        ↓
QQGateway
```

建议在 QQ 展示模块中建立以下边界：

- `PresentationPolicy`：只负责依据场景和数量选择模式；
- `PageRequest` 与 `Page`：校验页码并切分当前页数据；
- `render_card_listing`：渲染带封面的短列表；
- `render_structured_listing`：渲染分组后的中等列表；
- `render_compact_listing`：渲染高密度长列表；
- `render_detail_card`、`render_help`、`render_status`：处理固定场景；
- Markdown 转义与时间格式化函数：被所有渲染器复用。

领域模块不得返回 Markdown 字符串，也不得感知展示阈值。

## 5. 具体呈现

### 5.1 图文卡模式

QQ Markdown 不支持 CSS 或任意双栏布局，因此实际卡片采用原生纵向结构，不承诺视觉稿中的左右栏效果。

每页展示 5 部番剧：

```text
# 🌤 今日番剧
7 月 16 日 · 共 12 部

![番剧封面 #200px #112px](cover-url)
## 仙王的日常生活 第四季
**02:30 放送** · 第 4 话
Bangumi 405214

***
```

每条卡片包含封面、标题、预计放送时间、可用时的集数以及 Bangumi ID。数据陈旧提示统一放在页尾。

### 5.2 结构化 Markdown 模式

每页展示 15 条，按日期或星期分组：

```text
# 2026 年夏季番剧

## 周三 · 09/30
- **21:00** 女主角？圣女？不，我是杂役女仆！ `558064`
- **23:00** LV999 的村民 `579787`
```

Bangumi ID 作为次要信息显示，不再与主标题争夺视觉注意力。

### 5.3 紧凑文本模式

每页展示 30 条，每条只保留序号、标题、时间和 ID，不加载封面，不显示简介。

页尾统一显示 `第 N/M 页 · 共 X 部`。

### 5.4 固定场景

- `番剧详情`：封面、标题、首播日期、评分、话数、截断后的简介和常用操作按钮；
- `下次更新`：沿用详情卡结构，突出集数与预计放送时间，并保留资源实际上线时间免责声明；
- `帮助`：按查询、订阅和群管理分组，附常用命令按钮；
- `订阅/取消订阅`：使用成功、未变更或失败状态和明确的下一步操作。

## 6. 分页与按钮

- 图文卡每页 5 条；
- 结构化 Markdown 每页 15 条；
- 紧凑文本每页 30 条。

页面按需提供上一页、下一页和切换精简列表按钮。详情与搜索结果可以提供最多 5 个直接操作按钮，避免让按钮数量超过内容本身。

分页使用 QQ 回调按钮。按钮数据只携带查询类型、规范化查询条件、目标页码和目标展示模式，不携带或信任用户身份与权限。服务端收到回调后：

1. 立即确认交互，结束客户端加载状态；
2. 重新解析并校验按钮数据；
3. 重新执行只读查询；
4. 使用当前数据渲染目标页面；
5. 对越界页码返回边界页或明确错误。

订阅和管理操作仍经过原有身份与权限策略，不从按钮数据推断权限。

## 7. 降级与错误处理

每条富消息都必须包含等价、可读的纯文本兜底内容。

降级顺序为：

```text
图文 Markdown
  ↓ 单项封面缺失
无封面的单项卡片
  ↓ QQ 明确拒绝图片或 Markdown
结构化无图 Markdown
  ↓ QQ 明确拒绝 Markdown
纯文本
```

- 个别封面缺失只影响对应条目；
- 图片上传或拉取的明确错误触发一次无图降级；
- 超时、服务器错误等投递结果不确定时不自动重发，避免重复消息；
- Markdown 特殊字符统一转义；
- 简介按可见字符数截断，不截断到半个 Markdown 标记；
- 空结果、参数错误与权限错误不进入富消息降级链。

## 8. 数据流

以 `季度番剧 2026 夏` 为例：

1. 解析并校验命令参数；
2. 查询季度条目与放送时间；
3. 过滤 NSFW 条目；
4. `PresentationPolicy` 按实际数量选择模式；
5. 分页器取得当前页；
6. 渲染器同时生成 Markdown、纯文本兜底和分页按钮；
7. QQ 网关发送对应消息类型；
8. 若收到明确的格式拒绝，进入下一种降级格式；每种格式最多尝试一次，遇到超时或未知结果立即停止。

## 9. 测试设计

### 9.1 单元测试

- `1 / 20 / 21 / 50 / 51` 条的模式边界；
- NSFW 过滤后的模式选择；
- 三种模式的页大小和首、中、尾页；
- 非法、负数、零和越界页码；
- Markdown 特殊字符转义；
- 无封面、部分无封面和全部无封面；
- 帮助、详情、下次更新和订阅状态渲染；
- 纯文本兜底与 Markdown 内容语义一致。

### 9.2 合同与端到端测试

- Markdown 使用 `msg_type=2`；
- 回调按钮使用正确 action 类型和权限；
- 回调事件得到及时确认并能路由到目标页面；
- 明确图片错误只触发一次无图降级；
- 超时或未知结果不重复发送；
- 现有查询、订阅、权限和计划任务测试继续通过。

### 9.3 QQ 沙箱验收

在手机 QQ 与桌面 QQ 分别验证：

1. `今日番剧` 的封面、标题和放送时间层级清晰；
2. 20、21、50、51 条数据正确切换模式；
3. 翻页按钮无持续加载，首尾页行为正确；
4. 超长中日文标题不破坏布局；
5. 封面 URL 失效时仍可获得完整列表；
6. 帮助、详情和订阅状态均可快速扫读。

## 10. 完成标准

- 所有主动命令回复使用本设计对应的展示模式；
- 富消息始终具备完整纯文本兜底；
- 分页与交互确认稳定工作；
- 图片或 Markdown 失败不会丢失查询结果；
- 自动化测试、类型检查和静态检查全部通过；
- QQ 沙箱的手机端与桌面端验收通过。
